## Liquid Swords 为什么选择 UE5 开发《Samson》

[Unreal Engine | 查看原文](https://www.unrealengine.com/en-US/developer-interviews/taking-a-zero-nonsense-approach-to-game-development-why-liquid-swords-chose-ue5-for-samson)

斯德哥尔摩游戏工作室 Liquid Swords 正在开发一款名为《Samson》的硬核开放世界动作游戏。这篇文章深入解析了他们选择虚幻引擎5作为开发平台的核心原因，以及如何在项目中实现"零废话"（Zero-Nonsense）的高效开发理念。

### 关键技术：Nanite 让美术更自由

文章首先强调了 UE5 的 Nanite 虚拟几何体技术对他们项目的重要性。作为一款开放世界游戏，《Samson》需要处理大量高精度资产，而 Nanite 让他们能够在不牺牲性能的前提下使用电影级别的模型细节。这意味着美术团队可以更专注于创作，而不必过多考虑 LOD（细节层次）的繁琐设置。

### Lumen 实时光照革新工作流

其次，Lumen 全局光照系统大大简化了光照工作流程。传统的光照烘焙需要耗费大量时间，而 Lumen 提供的实时全局光照让关卡设计师能够即时看到光照效果，快速迭代场景氛围。对于追求"gritty（粗犷）"视觉风格的《Samson》来说，这种即时反馈至关重要。

### World Partition 管理大规模开放世界

文章还提到了 UE5 的 World Partition 系统如何帮助他们管理大规模开放世界。通过自动化的关卡流式加载，团队可以更专注于玩法设计，而不必手动管理复杂的加载边界。这种技术层面的自动化正是"零废话"理念的体现。

## UE5 游戏开发完整指南：Blueprint 与 C++ 的黄金平衡点

[Juego Studios | 查看原文](https://www.juegostudio.com/blog/top-unreal-game-development-companies-2025)

这是一篇全面的 UE5 开发指南，系统性地总结了使用虚幻引擎5进行游戏开发的最佳实践。文章的核心观点是：UE5 虽然减少了手动摩擦，但从项目第一天起就需要更强的资产结构化、光照策略和流式加载规范。

### 架构决策：什么时候用 C++？

在资产管线上方，作者强调了 Blueprint 的快速开发优势必须与 C++ 的系统控制相平衡。Blueprint 适合快速原型和简单逻辑，但在大规模项目中，关键系统（如角色控制、库存管理、存档系统）应当使用 C++ 实现以确保性能和稳定性。

文章提供了具体的判断标准：当逻辑需要在多个 Blueprint 中复用、或者涉及复杂的数据处理时，就应该考虑迁移到 C++。这种技术决策通常由技术策划在项目中后期主导。

### Nanite 的新挑战：内存管理

关于性能优化，文章特别指出了 UE5 中的新挑战。Nanite 虽然解决了多边形数量的限制，但带来了新的内存管理考量。开发者需要理解 Nanite 的屏幕空间细节级别机制，合理设置剔除距离，避免远距离物体的过度渲染。

## Json Level Builder：UE5 数据驱动关卡的技术突破

[Epic Developer Forums | 查看原文](https://forums.unrealengine.com/t/zedstudioz-json-level-builder/2702951)

Json Level Builder 是一个创新的 UE5 编辑器插件，它允许开发者将关卡布局导出为 JSON 文件并在运行时即时重新生成。这个工具对技术策划来说具有重要价值，因为它实现了关卡设计的"数据驱动化"。

### 解决传统工作流的痛点

传统的关卡设计流程中，关卡设计师在编辑器中手动摆放资产，这种方式虽然直观但存在几个问题：版本控制困难（二进制关卡文件）、协作冲突频繁、难以程序化生成变体。Json Level Builder 通过将关卡抽象为可读的 JSON 数据，解决了这些痛点。

### 运行时重新生成：PCG 的新可能

更强大的功能在于运行时重新生成。通过调用提供的 API，游戏可以在运行时读取 JSON 文件并动态生成关卡。这为程序化内容生成（PCG）打开了大门——技术策划可以编写算法生成不同的 JSON 布局，实现无限的关卡变体。

对于开放世界游戏或 Roguelike 游戏，这个工具特别有用。技术策划可以设计一套关卡"拼图"规则，然后让系统自动组合生成完整的关卡。
